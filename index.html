Avoid JVM Autoboxing Overhead

High-throughput system components must avoid standard Java Collection classes (like ArrayList<Integer>) in execution loops. The JVM boxes primitive int into heap-allocated Integer objects, creating pointer chasing and massive Garbage Collection overhead.

🛑 Boxing Trap (Slow Code)

// Heavy object allocations on every insertion
List<Long> timestamps = new ArrayList<>();
timestamps.add(currentTime); // Auto-boxing long -> Long

🟢 Zero-Boxing (Fast Code)

// Flat array avoids object allocations
long[] timestamps = new long[LIMIT];
timestamps[tail] = currentTime; // Zero allocations

Unsigned Pointer Arithmetic

When binary searching massive arrays (10,000,000+ items), calculating (low + high) / 2 can exceed Integer.MAX_VALUE, wrapping the sum to negative values and throwing index out of bounds exceptions.

🟢 Overflow-Safe Midpoint Shift

// Correct: Unsigned right shift operator >>>
int mid = (low + high) >>> 1;

Why it works:

The >>> operator shifts the bit representation to the right, inserting a zero at the most significant bit. Even if low + high overflows and becomes a negative number, the unsigned shift treats it as positive, successfully obtaining the correct midpoint.
