# Pattern Blueprint: Sliding Window

## 1. System Design Mapping
* **Macro System Component:** API Gateway Rate Limiter (Sliding Window Counter)
* **How it leverages this DSA Pattern:**
  We must track request arrivals over a dynamic rolling time window (e.g., 60 seconds). A circular queue stores request timestamps. When a new request arrives, any timestamps older than `current_time - window_size` are evicted from the queue. If the remaining count is within the rate limit, the request is allowed and the current timestamp is appended.
* **Data Flow Architecture:**
  ```text
  [Incoming Request] ──> [Evict Outdated Timestamps from Head] ──> [Check Queue Size] 
                                                                           │
                                                                           ├─> Over Limit ──> Reject
                                                                           └─> Under Limit ─> Add to Tail & Accept
  ```

## 2. High-Yield Performance Tricks (Java Specific)
* **Memory & Collection Strategy:** 
  Avoid standard `java.util.ArrayDeque<Long>` or `java.util.LinkedList<Long>` due to pointer chasing and object boxing overhead. Instead, implement a custom circular buffer using a primitive `long[]` array.
* **Time/Space Constraints:** 
  * Time Complexity: $O(K)$ where $K$ is the number of stale entries to evict. On average, $O(1)$ per request.
  * Space Complexity: $O(L)$ where $L$ is the rate limit (maximum size of the buffer).
* **Garbage Collection (GC) Impact:**
  Zero allocation in the critical path. The primitive array is pre-allocated upon startup, generating no garbage collections during request execution.

## 3. The Core Structural Trick (Mental Model)
> Evict elements from the head while the condition is violated (timestamp < threshold), then append the new item. The window size represents the size of the active elements.
