# Pattern Blueprint: Graph BFS/DFS

## 1. System Design Mapping
* **Macro System Component:** Microservice Call Graph Trace Optimizer (Distributed Tracing Analysis)
* **How it leverages this DSA Pattern:**
  Distributed systems track transactions as directed acyclic graphs representing RPC calls between services. To trace cycles, compute network latency bottlenecks, or find critical execution paths, we build call graph representations and traverse them. BFS evaluates parallel fan-out layers, and DFS isolates recursive critical pathways.
* **Data Flow Architecture:**
  ```text
  Trace Spans ──> [Build Adjacency Graph] ──> [DFS: Critical Path Latency]
                                          └──> [BFS: Hop Count/Fan-Out Breadth]
  ```

## 2. High-Yield Performance Tricks (Java Specific)
* **Memory & Collection Strategy:**
  Avoid allocating `Node` objects with `List<Node> neighbors` references. For high-volume trace analysis, represent graphs using flat arrays: a concatenated adjacency array `int[] edges` and an offset index array `int[] offsets` (Compressed Sparse Row representation).
* **Time/Space Constraints:**
  * Time Complexity: $O(V + E)$ where $V$ is vertices (services) and $E$ is edges (RPC calls).
  * Space Complexity: $O(V + E)$ storage, $O(V)$ stack/queue depth.
* **Garbage Collection (GC) Impact:**
  Flat array representation prevents pointer dereferencing and eliminates heap-allocation GC spikes.

## 3. The Core Structural Trick (Mental Model)
> BFS finds shortest paths and layer boundaries using a FIFO queue; DFS searches deep pathways and detects cycles using a LIFO stack (or recursion).
