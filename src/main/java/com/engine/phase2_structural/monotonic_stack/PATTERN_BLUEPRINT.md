# Pattern Blueprint: Monotonic Stack

## 1. System Design Mapping
* **Macro System Component:** Telemetry Metric Stream Monitoring (Next Max Spike Duration)
* **How it leverages this DSA Pattern:**
  We monitor high-frequency metric data streams (e.g., CPU load updates). To track the latency or duration until a resource usage record is broken, we store streaming indices in a stack. If a incoming value exceeds the metric at the index on top of the stack, we resolve the "next maximum spike" for that entry by calculating the index delta.
* **Data Flow Architecture:**
  ```text
  Incoming Metric stream: 70 -> 60 -> 55 -> 80
  
  Stack: [70_idx] ──> [70_idx, 60_idx] ──> [70_idx, 60_idx, 55_idx]
  
  On incoming 80:
  - 80 > 55 -> Pop 55, calc delta.
  - 80 > 60 -> Pop 60, calc delta.
  - 80 > 70 -> Pop 70, calc delta.
  Stack: [80_idx]
  ```

## 2. High-Yield Performance Tricks (Java Specific)
* **Memory & Collection Strategy:**
  Avoid object wrapper types like `java.util.Stack<Integer>` or `java.util.ArrayDeque<Integer>` which box coordinates. Use a pre-allocated primitive array `int[]` as the stack and a simple pointer variable `top` (initialized to `-1`) to track the stack head.
* **Time/Space Constraints:**
  * Time Complexity: $O(N)$ total (each element is pushed and popped at most once).
  * Space Complexity: $O(N)$ in the worst case (monotonically decreasing stream).
* **Garbage Collection (GC) Impact:**
  Zero GC overhead. Creating the stack as an array pre-allocated to the batch size completely avoids garbage collection pauses.

## 3. The Core Structural Trick (Mental Model)
> Maintain stack monotonicity by popping elements that are smaller than the incoming value, which resolves those elements' next-greater query.
